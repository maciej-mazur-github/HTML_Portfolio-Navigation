Adventure Game
Difficulty

Simulation of a "maze runner" style game. HashMaps have been used in a pretty creative way. There is an immutable class Location of which each object contains its own... HashMap with exits, mapping direction string IDs (S, N, W, E and Q) with their corresponding locationID numbers, e.g., if you are currently in the location #1 and if you choose to go North, then you will reach the location #5 (please see the attached .png file in the repository). There is also another class used, AdventureGame, that contains 2 other HashMaps – ‘locations’ and ‘directionsVocabulary’. HashMap "locations" contains a mapping between each location's integer ID and the corresponding object of class Location (the latter one basically binding information on given location's possible exit IDs and to which locations these exits lead). HashMap ‘directionsVocabulary’ plays a role of mapping keywords representing directions of the world (SOUTH, NORTH etc.) with their 1-character IDs (S, N etc.). The usage of toUpperCase() and splitting user's string input (when asked in which direction they want to go) allows the program to properly process not only short, case-sensitive answers like S, N, but also the answers like "I now need to go south" (as this answer contains the keyword "south"). Both HashMaps 'locations' and 'directionsVocabulary' are fully initialized by AdventureGame's constructor. Each location in 'locations' HashMap contains Q->0 mapping which allows the user to break program's main infinite loop and hence to end the program whenever 'Q' or the keyword 'quit' is entered.Read more

Theatre Seat Reservation System
Difficulty

Pretty expanded program simulating the theatre seat reservation system. The inner class Seat is used that implements Comparable interface and overrides its compareTo() method..., later used by Collections.binarySearch(). The row number and seats per row number are both set up in a constructor call. Also, the constructor creates all the seats in 'seats' ArrayList in ascending order (by seat number, which actually is a String e.g. H05, A12, E09 etc.), setting their prices as 7$, 12$ or 14$ according to their location relative to the margins (being ¼ of row count and seat per row count respectively). You can also sort all the seats by price order. In that case the interface Comparator variable is used and an anonymous class to override compare() method, then Collections.sort() method is used. You can also shuffle all the seats with a method based on Collections.shuffle() and then, if you prefer, sort them all again by name in ascending order using a method based on Selection Sort algorithm. You can show all the seats, including their number, price and reservation status. Method showSeats() uses a String 'builder' to construct a desired presentation. When reserving a seat, the validity of the number entered by the user is verified first. Lower case letters are replaced with uppercase and 1-digit seat numbers (in a row) have leading zero added. Then, if the ArrayList is sorted by seat number in ascending order, 3 types of searching the desired seat are available - Brute Force, Binary Search (based on Collections.binarySearch()) and manually implemented Binary Search. Brute Force and manually implemented Binary Search also print dots to let the user be aware of the time complexity of both algorithms. You can also cancel the reservation of any seat. The only public method run() is based on an infinite loop that can be broken only by choosing zero by the user.Read more

Binary Search Tree Implementation
Difficulty

Pretty sophisticated implementation of Binary Search Tree (BST). Abstract class Item and interface List have been used. You can add items using either a recursive or... non-recursive method. In both cases you are restricted from adding an existing item. The removal of the item is only available in a recursive form. All scenarios of removal are covered - when a found and to-be-removed item has no subtree whatsoever, when it has only the left subtree, when it only has a right subtree and finally when it has both left and right subtrees. In the latter scenario finding the smallest value item in the right subtree has been used to re-build the current structure of the tree. The code is guarded from the attempt to remove a non-existent item. You can also check whether a given item already exists in the BST or not, using recursive method. Variables foundSearchedItem and lastFoundItem have been used to let all the methods use them any time. Traversing the list is available in 3 ways:

1. Recursively.
2. Using Stack and no recursion.
3. Using no Stack and no recursion. Read more

Pretty creative and optimized approach to resolving the Flour Pack Problem quoted below...:

"Write a method named canPack with three parameters of type int named bigCount, smallCount, and goal.
The parameter bigCount represents the count of big flour bags (5 kilos each). The parameter smallCount represents the count of small flour bags (1 kilo each). The parameter goal represents the goal amount of kilos of flour needed to assemble a package. Therefore, the sum of the kilos of bigCount and smallCount must be at least equal to the value of goal. The method should return true if it is possible to make a package with goal kilos of flour. If the sum is greater than goal, ensure that only full bags are used towards the goal amount.

For example, if goal = 9, bigCount = 2, and smallCount = 0, the method should return false since each big bag is 5 kilos and cannot be divided. However, if goal = 9, bigCount = 1, and smallCount = 5, the method should return true because of 1 full bigCount bag and 4 full smallCount bags equal goal, and it's okay if there are additional bags left over.

If any of the parameters are negative, return false."Read more
